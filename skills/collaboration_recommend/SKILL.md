# 技能：科研协作推荐

## 技能名称
`collaboration_recommend` — 科研协作智能推荐

## 核心目标
分析团队中所有学生的能力矩阵，通过计算互补性得分，为导师推荐最佳科研协作组合。帮助导师在组建项目团队或安排合作时，找到技能互补、方向契合的学生搭配，最大化团队产出。

## 触发条件
当用户发出以下类型的请求时激活本技能：
- "帮我推荐几组合适的协作搭配"
- "哪些同学适合一起做这个项目"
- "看看谁和某某同学互补性最强"
- "组建一个三人小组做 XX 方向"
- 包含关键词：协作推荐、组队建议、互补性、合作搭配、项目组队

## 输入格式
```json
{
  "student_ids": ["string (可选，指定候选学生列表)"],
  "research_area": "string (可选，指定项目研究方向)",
  "team_size": "int (可选，默认2人)",
  "top_k": "int (可选，推荐组数，默认5)",
  "required_skills": ["string (可选，项目必需的能力维度)"],
  "exclude_pairs": [["string", "string"] , "(可选，排除已有的搭配)"]
}
```

## 输出格式
```json
{
  "recommendations": [
    {
      "rank": 1,
      "students": [
        {"id": "s001", "name": "张三", "strengths": ["代码实现", "实验设计"]},
        {"id": "s005", "name": "李四", "strengths": ["学术写作", "论文阅读"]}
      ],
      "complementarity_score": 0.87,
```

## 执行步骤

### 第零步（必须）：获取真实数据上下文
**如果用户提到了具体学生姓名**（如"张旭华老师跟谁合作"），**必须先调用 `fetch_collaboration_context(student_name)`**，从 CoEvo 系统获取：
- 所参与的项目及其季度目标、近期目标
- 在历次会议中遇到的 key_blockers（核心障碍）
- 导师给出的建议和 AI 会议洞察
- CoEvo 系统已生成的历史协作推荐
- OpenClaw 能力评分（如已同步）

**这些真实数据是推荐的核心依据，不可跳过，不可用占位符代替。**

### 第一步：构建能力矩阵
调用 `build_skill_matrix(students)` 从数据库拉取所有候选学生的最新能力评分。如第零步已获取目标学生的能力评分，直接使用。

### 第二步：计算互补性得分
对每一对学生，调用 `calculate_complementarity(student_a, student_b)` 计算互补性。
**结合第零步数据深化评估**：
- **障碍互补**：对方强项是否覆盖目标学生的 key_blockers（最重要）
- **目标协同**：双方的项目目标是否有协同空间
- **技能互补度**：一方的强项恰好补足另一方的弱项
- **历史合作信号**：CoEvo 历史推荐中是否有类似搭配

### 第三步：排序与筛选
根据互补性得分排序，满足必需技能覆盖，排除不适合搭配，确保组合多样性。

### 第四步：生成推荐理由
**必须引用真实数据**：
- "A 在 [某项目] 中遇到 [具体 blocker]，而 B 在该方面恰好有优势"
- 引用导师建议或会议洞察中的具体内容
- 如有历史协作记录，说明与 CoEvo 分析的一致性
- **严禁**生成"获取到这些数据后我将为您计算"等占位性文本

### 第五步：持久化推荐结果
调用 `save_recommendations(recommendations)` 保存到数据库。

## 注意事项

### 公平性
- 确保每位学生都有被推荐的机会，不应总是推荐同几位"明星学生"
- 考虑学生的工作负载，避免给已经忙碌的学生安排过多合作

### 隐私保护
- 不在推荐理由中暴露具体分数，只描述相对优势
- 如 "张三在代码实现方面较为突出" 而非 "张三代码实现得分95分"

### 语言风格
- 使用中文，语气客观专业
- 推荐理由应具体而非空泛
- 同时说明搭配的优势和可能需要注意的协作要点

### 算法透明度
- 每条推荐附带互补性得分和覆盖维度
- 导师可以理解推荐逻辑，而非黑盒推荐
